# Skill: Task Decomposition

## Purpose

Break a structured specification into implementation-ready backlog items.

## Use this skill when

- a specification already exists
- the work must be split into reviewable units
- backlog structure is needed before sprint planning

## Inputs

- structured feature specification
- optional architectural notes

## Output contract

Produce:

- parent work item
- feature tasks
- optional sub-tasks
- dependencies
- implementation sequence notes

## Rules

- prefer small reviewable tasks
- isolate backend, frontend, database, QA, and docs work where useful
- do not create meaningless micro-tasks
- keep each task understandable on its own

## Escalate when

- decomposition depends on unresolved architecture
- cross-team dependency is unclear
- the work cannot be split into incremental units
